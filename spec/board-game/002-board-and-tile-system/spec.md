**기능 브랜치**: `002-board-and-tile-system`  
**작성일**: 2026-04-20  
**상태**: Draft  
**입력 설명**: "주사위 기반 부동산 보드게임에서 플레이어가 이동할 수 있는 보드와 각 칸의 타입, 속성, 효과를 정의한다."

---

## 1. 기능 개요 *(필수)*

### 기능 설명

이 기능은 게임에서 사용하는 보드 구조를 정의한다.

보드는 순환형 칸 배열로 구성되며, 플레이어는 주사위 결과에 따라 현재 위치에서 앞으로 이동한다.

각 칸은 `START`, `CITY`, `CHANCE`, `ISLAND`, `TRAVEL`, `TAX` 타입 중 하나를 가진다. 도시 칸은 구매, 소유, 업그레이드, 통행료 지불 대상이 될 수 있다. 특수 칸은 별도 효과를 발생시킨다.

### 해결하려는 문제

보드 구조가 명확하지 않으면 이동, 구매, 통행료, 특수 효과, 승패 판정 로직이 일관되게 동작할 수 없다.

### 기대 효과

- 모든 게임 로직이 동일한 보드 정의를 기준으로 동작한다.
- 보드 크기와 칸 타입을 데이터 기반으로 변경할 수 있다.
- 테스트 시 특정 칸에 플레이어를 배치하여 구매, 이동, 효과 처리를 독립 검증할 수 있다.

### 완료 조건

- 32칸 보드를 생성할 수 있다.
- 각 칸은 고유한 `tileId`와 `index`를 가진다.
- 각 칸은 하나의 `tileType`을 가진다.
- 도시 칸은 가격, 통행료, 업그레이드 비용을 가진다.
- 보드는 순환 구조로 동작한다.

## 2. 기능 범위 *(필수)*

### 포함 범위 (In Scope)

- 보드 초기화
- 칸 타입 정의
- 도시 칸 데이터 정의
- 특수 칸 데이터 정의
- 위치 인덱스 기반 칸 조회
- 순환형 보드 이동을 위한 인덱스 계산

### 제외 범위 (Out of Scope)

- 실제 플레이어 이동 처리
- 도시 구매 처리
- 통행료 지불 처리
- 특수 칸 효과 실행
- UI 렌더링
- 맵 스킨, 캐릭터, 애니메이션

### MVP 범위

본 기능의 MVP는 32개 칸으로 구성된 기본 보드를 생성하고, index로 칸 정보를 조회할 수 있는 상태이다.

기본 보드 구성:

- 0번: `START`
- 3, 11, 19, 27번: `CHANCE`
- 8번: `ISLAND`
- 16번: `TRAVEL`
- 24번: `TAX`
- 나머지 칸: `CITY`

MVP에서는 `EMPTY` 칸을 두지 않는다.

## 3. 기능 시나리오 *(필수)*

### 시나리오 1 - 기본 보드 생성 (우선순위: P1)

시스템은 게임 시작 시 기본 보드를 생성한다.

**행위자**: 시스템

**왜 중요한가**: 보드는 모든 이동과 상호작용의 기준이다.

**독립 테스트 방법**: 보드 생성 함수를 호출하고 칸 개수, index, tileType을 검증한다.

**인수 조건**:

1. **Given** 게임 세션이 생성될 때, **When** 보드를 초기화하면, **Then** 32개의 `Tile`이 생성되어야 한다.
2. **Given** 보드가 생성되었을 때, **When** 0번 칸을 조회하면, **Then** `tileType`은 `START`여야 한다.
3. **Given** 보드가 생성되었을 때, **When** 3번 칸을 조회하면, **Then** `tileType`은 `CHANCE`여야 한다.

---

### 시나리오 2 - 도시 칸 데이터 조회 (우선순위: P1)

시스템은 도시 칸의 구매 가격, 기본 통행료, 업그레이드 비용을 조회한다.

**행위자**: 시스템

**왜 중요한가**: 도시 구매와 통행료 계산은 도시 칸 데이터에 의존한다.

**독립 테스트 방법**: `CITY` 타입 칸을 조회하고 `purchasePrice`, `baseToll`, `upgradeCosts` 필드가 존재하는지 확인한다.

**인수 조건**:

1. **Given** `CITY` 타입 칸이 있을 때, **When** 칸 정보를 조회하면, **Then** `purchasePrice`, `baseToll`, `upgradeCosts`가 반환되어야 한다.
2. **Given** `START` 타입 칸이 있을 때, **When** 도시 속성을 조회하면, **Then** 도시 속성은 `null` 또는 `undefined`여야 한다.

---

### 시나리오 3 - 순환형 위치 계산 (우선순위: P1)

보드는 마지막 칸 이후 다시 0번 칸으로 이어진다.

**행위자**: 시스템

**왜 중요한가**: 주사위 이동 시 시작 지점을 통과하면 보너스 지급 등의 처리가 필요하다.

**독립 테스트 방법**: 현재 위치와 이동 칸 수를 입력하여 도착 위치와 START 통과 여부를 검증한다.

**인수 조건**:

1. **Given** 현재 위치가 30이고 주사위 값이 4일 때, **When** 이동 위치를 계산하면, **Then** 도착 위치는 2여야 한다.
2. **Given** 현재 위치가 30이고 주사위 값이 4일 때, **When** 이동 위치를 계산하면, **Then** START 통과 여부는 `true`여야 한다.
3. **Given** 현재 위치가 5이고 주사위 값이 3일 때, **When** 이동 위치를 계산하면, **Then** 도착 위치는 8이고 START 통과 여부는 `false`여야 한다.

## 4. 예외 상황 및 경계 조건 *(필수)*

- 보드 크기가 1보다 작으면 `BOARD_SIZE_INVALID`를 반환한다.
- 칸 index가 0보다 작거나 보드 크기 이상이면 `TILE_INDEX_INVALID`를 반환한다.
- `CITY` 칸에 가격 정보가 없으면 `CITY_DATA_INVALID`를 반환한다.
- 특수 칸에 효과 타입이 없으면 `SPECIAL_TILE_DATA_INVALID`를 반환한다.
- 동일 index를 가진 칸이 중복되면 보드 생성에 실패한다.
- tileId가 중복되면 보드 생성에 실패한다.
- 이동 칸 수가 0 이하이면 `MOVE_DISTANCE_INVALID`를 반환한다.

## 5. 요구사항 *(필수)*

### 기능 요구사항

- **FR-001**: 시스템은 32칸 기본 보드를 생성해야 한다.
- **FR-002**: 시스템은 각 칸에 고유한 `tileId`를 부여해야 한다.
- **FR-003**: 시스템은 각 칸에 0부터 31까지의 `index`를 부여해야 한다.
- **FR-004**: 시스템은 각 칸의 타입을 `START`, `CITY`, `CHANCE`, `ISLAND`, `TRAVEL`, `TAX` 중 하나로 지정해야 한다.
- **FR-005**: 시스템은 index 기준으로 칸 정보를 조회할 수 있어야 한다.
- **FR-006**: 시스템은 `CITY` 칸에 구매 가격, 기본 통행료, 업그레이드 비용을 설정해야 한다.
- **FR-007**: 시스템은 순환형 보드 이동 결과를 계산할 수 있어야 한다.
- **FR-008**: 시스템은 START 통과 여부를 계산할 수 있어야 한다.

### 비기능 요구사항

- **NFR-001**: 칸 조회는 50ms 이내에 완료되어야 한다.
- **NFR-002**: 보드 정의는 코드 하드코딩 대신 JSON 또는 설정 객체로 분리 가능해야 한다.
- **NFR-003**: 보드 데이터는 테스트 환경에서 쉽게 교체 가능해야 한다.

## 6. 핵심 엔터티 *(데이터 관련 기능이면 필수)*

- **Board**: 게임 보드를 의미한다. 주요 속성은 `boardId`, `size`, `tiles`이다.
- **Tile**: 보드의 한 칸을 의미한다. 주요 속성은 `tileId`, `index`, `name`, `tileType`이다.
- **CityTileData**: 도시 칸의 경제 정보를 의미한다. 주요 속성은 `purchasePrice`, `baseToll`, `upgradeCosts`, `colorGroup`이다.
- **SpecialTileData**: 특수 칸의 효과 정보를 의미한다. 주요 속성은 `effectType`, `description`이다.
- **MoveCalculationResult**: 이동 계산 결과를 의미한다. 주요 속성은 `fromIndex`, `toIndex`, `passedStart`이다.

## 8. 성공 기준 *(필수)*

### 측정 가능한 결과

- **SC-001**: 보드 생성은 100ms 이내에 완료되어야 한다.

## 9. 가정사항 *(필수 권장)*

- MVP 보드 크기는 32칸으로 고정한다.
- 보드는 원형 순환 구조라고 가정한다.
- 모든 플레이어는 동일한 보드를 공유한다.
- MVP에서는 맵 스킨 변경을 지원하지 않는다.
- MVP에서는 도시 이름을 임의의 독자 이름으로 사용한다.
