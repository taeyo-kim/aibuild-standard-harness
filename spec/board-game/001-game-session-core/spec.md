# 기능 명세서: 게임 세션 생성 및 상태 관리

**기능 브랜치**: `001-game-session-core`  
**작성일**: 2026-04-20  
**상태**: Draft  
**입력 설명**: "2~4명의 플레이어가 참여하는 주사위 기반 부동산 보드게임 세션을 생성하고, 게임 상태를 일관되게 관리한다."

---

## 1. 기능 개요 *(필수)*

### 기능 설명

이 기능은 주사위 기반 부동산 보드게임의 한 판을 생성하고 관리한다.

게임 세션은 플레이어 목록, 보드 정보, 현재 턴, 플레이어별 자산, 위치, 보유 도시, 게임 진행 상태를 포함한다. 이후 주사위 굴림, 이동, 도시 구매, 통행료 지불, 특수 칸 효과, 승패 판정 기능은 모두 이 게임 세션 상태를 기준으로 동작한다.

### 해결하려는 문제

게임 진행 중 여러 기능이 각각 상태를 수정하면 현재 턴, 플레이어 위치, 자산, 소유권, 게임 종료 여부가 불일치할 수 있다.

따라서 모든 게임 로직은 하나의 명확한 `GameSession` 상태를 기준으로 동작해야 한다.

### 기대 효과

- 게임 한 판의 상태를 일관되게 관리할 수 있다.
- 각 기능이 동일한 게임 상태 모델을 기준으로 구현된다.
- 테스트 시 특정 게임 상태를 만들어 턴, 이동, 구매, 승패 판정을 독립적으로 검증할 수 있다.
- 향후 온라인 멀티플레이로 확장할 때 서버 권위형 상태 관리 구조로 전환하기 쉽다.

### 완료 조건

- 2~4명의 플레이어로 게임 세션을 생성할 수 있다.
- 각 플레이어는 초기 자산, 시작 위치, 빈 소유 도시 목록을 가진다.
- 게임은 `WAITING`, `IN_PROGRESS`, `FINISHED` 상태를 가진다.
- 게임 시작 시 첫 번째 턴 플레이어가 결정된다.
- 게임 상태는 조회 가능해야 한다.
- 게임 종료 후에는 추가 액션이 거부되어야 한다.

## 2. 기능 범위 *(필수)*

### 포함 범위 (In Scope)

- 게임 세션 생성
- 플레이어 등록
- 초기 자산 지급
- 초기 위치 설정
- 게임 시작 처리
- 현재 턴 플레이어 관리
- 게임 상태 조회
- 게임 종료 상태 관리
- 게임 이벤트 로그 기록

### 제외 범위 (Out of Scope)

- 실제 주사위 굴림 로직
- 보드 칸 이동 로직
- 도시 구매 및 통행료 계산
- 특수 칸 효과
- 실시간 네트워크 동기화
- 채팅, 이모티콘, 랭킹, 친구 초대
- 실제 결제, 재화 충전, 광고

### MVP 범위

본 기능의 MVP는 다음 조건을 만족하는 상태이다.

- 2~4명의 플레이어로 게임 세션을 생성할 수 있다.
- 게임 시작 시 모든 플레이어가 시작 위치 0번 칸에 배치된다.
- 모든 플레이어는 초기 자산 3,000,000 골드를 가진다.
- 첫 번째 플레이어부터 턴이 시작된다.
- 현재 게임 상태를 JSON 형태로 조회할 수 있다.

## 3. 기능 시나리오 *(필수)*

### 시나리오 1 - 게임 세션 생성 (우선순위: P1)

사용자는 2~4명의 플레이어 정보를 입력하여 새로운 게임 세션을 생성한다.

**행위자**: 사용자 / 시스템

**왜 중요한가**: 게임의 모든 기능은 세션이 존재해야만 동작한다.

**독립 테스트 방법**: 플레이어 2명, 3명, 4명을 입력해 세션 생성 API 또는 함수를 호출하고 반환된 `GameSession` 객체를 검증한다.

**인수 조건**:

1. **Given** 플레이어 2명의 이름이 입력되었을 때, **When** 게임 세션을 생성하면, **Then** `WAITING` 상태의 `GameSession`이 생성되어야 한다.
2. **Given** 플레이어 1명만 입력되었을 때, **When** 게임 세션을 생성하면, **Then** 생성이 거부되고 `PLAYER_COUNT_INVALID` 에러가 반환되어야 한다.
3. **Given** 플레이어 5명이 입력되었을 때, **When** 게임 세션을 생성하면, **Then** 생성이 거부되고 `PLAYER_COUNT_INVALID` 에러가 반환되어야 한다.

---

### 시나리오 2 - 게임 시작 (우선순위: P1)

생성된 세션을 실제 진행 상태로 변경한다.

**행위자**: 사용자 / 시스템

**왜 중요한가**: 게임 시작 전에는 턴 액션, 주사위 굴림, 구매 액션이 실행되면 안 된다.

**독립 테스트 방법**: `WAITING` 상태의 세션에서 게임 시작 함수를 호출하고 상태, 현재 턴, 초기 자산을 검증한다.

**인수 조건**:

1. **Given** `WAITING` 상태의 세션이 있을 때, **When** 게임을 시작하면, **Then** 세션 상태는 `IN_PROGRESS`가 되어야 한다.
2. **Given** 게임이 시작되었을 때, **When** 플레이어 상태를 조회하면, **Then** 모든 플레이어의 위치는 0이고 자산은 3,000,000이어야 한다.
3. **Given** 게임이 시작되었을 때, **When** 현재 턴을 조회하면, **Then** 첫 번째 플레이어가 현재 턴 플레이어여야 한다.

---

### 시나리오 3 - 게임 상태 조회 (우선순위: P1)

클라이언트 또는 테스트 코드는 현재 게임 상태를 조회한다.

**행위자**: 사용자 / 시스템

**왜 중요한가**: UI 렌더링, 테스트, 턴 처리, 승패 판정은 모두 현재 상태 조회를 필요로 한다.

**독립 테스트 방법**: 세션 생성 후 상태 조회 함수를 호출하고 반환 데이터 구조를 검증한다.

**인수 조건**:

1. **Given** 진행 중인 게임이 있을 때, **When** 게임 상태를 조회하면, **Then** 세션 ID, 상태, 플레이어 목록, 현재 턴, 보드 상태가 반환되어야 한다.
2. **Given** 존재하지 않는 세션 ID가 입력되었을 때, **When** 게임 상태를 조회하면, **Then** `SESSION_NOT_FOUND` 에러가 반환되어야 한다.

## 4. 예외 상황 및 경계 조건 *(필수)*

- 플레이어 수가 2명 미만이면 `PLAYER_COUNT_INVALID`를 반환한다.
- 플레이어 수가 4명을 초과하면 `PLAYER_COUNT_INVALID`를 반환한다.
- 플레이어 이름이 비어 있으면 기본 이름 `Player {index}`를 부여한다.
- 중복된 플레이어 이름은 허용하되 내부 `playerId`는 고유해야 한다.
- 이미 시작된 게임에 대해 다시 시작 요청이 오면 `GAME_ALREADY_STARTED`를 반환한다.
- 종료된 게임에 대해 시작 요청이 오면 `GAME_ALREADY_FINISHED`를 반환한다.
- 존재하지 않는 세션 조회 시 `SESSION_NOT_FOUND`를 반환한다.
- 게임 종료 상태에서는 턴 액션, 구매 액션, 이동 액션을 모두 거부한다.

## 5. 요구사항 *(필수)*

### 기능 요구사항

- **FR-001**: 시스템은 2~4명의 플레이어로 게임 세션을 생성해야 한다.
- **FR-002**: 시스템은 각 플레이어에게 고유한 `playerId`를 부여해야 한다.
- **FR-003**: 시스템은 생성된 세션에 고유한 `sessionId`를 부여해야 한다.
- **FR-004**: 시스템은 게임 상태를 `WAITING`, `IN_PROGRESS`, `FINISHED` 중 하나로 관리해야 한다.
- **FR-005**: 시스템은 게임 시작 시 모든 플레이어에게 초기 자산 3,000,000 골드를 지급해야 한다.
- **FR-006**: 시스템은 게임 시작 시 모든 플레이어의 위치를 0번 칸으로 설정해야 한다.
- **FR-007**: 시스템은 현재 턴 플레이어를 추적해야 한다.
- **FR-008**: 시스템은 게임 상태를 조회할 수 있어야 한다.
- **FR-009**: 시스템은 종료된 게임에 대한 추가 액션을 거부해야 한다.
- **FR-010**: 시스템은 모든 주요 상태 변경을 이벤트 로그로 기록해야 한다.

### 비기능 요구사항

- **NFR-001**: 게임 상태 조회는 200ms 이내에 완료되어야 한다.
- **NFR-002**: 게임 세션 생성은 500ms 이내에 완료되어야 한다.
- **NFR-003**: 동일 세션 내 상태 변경은 순차적으로 처리되어야 한다.
- **NFR-004**: 게임 상태는 디버깅 가능한 JSON 구조로 직렬화 가능해야 한다.
- **NFR-005**: 모든 상태 변경 이벤트는 재현 가능한 형태로 기록되어야 한다.

## 6. 핵심 엔터티 *(데이터 관련 기능이면 필수)*

- **GameSession**: 게임 한 판을 의미한다. 주요 속성은 `sessionId`, `status`, `players`, `board`, `currentTurnPlayerId`, `turnNumber`, `winnerId`, `eventLogs`이다.
- **Player**: 게임에 참여하는 사용자 또는 봇을 의미한다. 주요 속성은 `playerId`, `name`, `money`, `position`, `ownedPropertyIds`, `status`, `isBot`이다.
- **GameStatus**: 게임 진행 상태를 의미한다. 값은 `WAITING`, `IN_PROGRESS`, `FINISHED`이다.
- **EventLog**: 게임 중 발생한 상태 변경 기록이다. 주요 속성은 `eventId`, `turnNumber`, `playerId`, `eventType`, `payload`, `createdAt`이다.

## 8. 성공 기준 *(필수)*

### 측정 가능한 결과

- **SC-001**: 게임 세션 생성은 500ms 이내에 완료되어야 한다.
- **SC-002**: 정상 입력의 99% 이상이 첫 시도에서 성공적으로 세션을 생성해야 한다.
- **SC-003**: 잘못된 플레이어 수 입력은 100% 에러로 처리되어야 한다.
- **SC-004**: 게임 시작 후 모든 플레이어의 초기 상태는 테스트 케이스 기준 100% 일치해야 한다.
- **SC-005**: 상태 조회 결과는 클라이언트 UI 렌더링에 필요한 필드를 100% 포함해야 한다.

## 9. 가정사항 *(필수 권장)*

- 플레이어는 이미 인증되어 있다고 가정한다.
- MVP에서는 게스트 플레이어 또는 봇 플레이어를 허용한다.
- MVP에서는 서버 저장소 대신 메모리 저장소를 사용할 수 있다.
- MVP에서는 실시간 멀티플레이가 아닌 단일 프로세스 게임 진행을 허용한다.
- 모든 금액 단위는 `gold`로 통일한다.
